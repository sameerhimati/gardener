# Gardener — PRD & Build Plan
**Harness Engineering Hack · One-day build · Demo at 4:30 PM**

> Working title: **Gardener** (alt: Vault, Homeostasis, Tend). An agent OS whose memory maintains itself.

---

## 1. One-liner

**Every personal agent rots — its memory fills with stale facts, contradictions, and dead ends. Gardener is a personal agent that logs every interaction and tool call to ClickHouse, then runs a background "lint agent" that continuously audits and rewrites its own memory vault. The more you use it, the cleaner and smarter it gets.**

Secondary hook: the agent has no fixed UI. For each user need, it generates a micro-app on the fly (gym tracker, reading rack, email triage) via generative UI.

- **Visceral demo:** Before/after diff of the memory vault is something judges can literally watch.
- **Deep sponsor usage (not checkbox):**

## 3. Problem

Personal AI agents accumulate context: preferences, facts, past decisions, tool-call history. Today that context either (a) doesn't persist, (b) persists but rots (stale jobs, moved cities, contradicted preferences), or (c) requires the user to manually curate it. Memory rot silently degrades every future response. No consumer or dev product treats **memory hygiene as an automated, observable process**.

## 4. Core insight

Agent observability data (traces, tool calls, outcomes) is not just for debugging — it is the **raw material for memory maintenance**. If every interaction is logged in a queryable analytical store, a second agent can mine it for:
- Contradictions ("user said they're vegetarian on Tue, ordered steak Fri")
- Staleness (facts not referenced or reaffirmed in N days)
- Failure patterns (tool calls that repeatedly error → rewrite the skill/instructions)
- Redundancy (duplicate or near-duplicate memory entries → merge)
- Gaps (questions the agent repeatedly couldn't answer → flag for user)

This is **lint for memory**: deterministic rules + LLM judgment, run automatically, producing diffs.

## 5. Product shape (hackathon scope)

A web app (deployed on Render) with three panes:

1. **Chat / Command bar** — talk to your agent.
2. **Generated app canvas** — Thesys/OpenUI renders whatever micro-app the current intent needs (tracker, list, form, dashboard).
3. **The Garden (memory view)** — the user's vault as markdown files, with a live "lint feed": each lint run shows findings and proposed diffs; user can auto-accept or review.

### Primary user flows (build exactly these, nothing more)

**Flow A — Use the agent, watch memory grow**
1. User: "I want to track my gym progress."
2. Agent generates a workout-tracker micro-app (Thesys), user logs a workout.
3. Every message + tool call + UI generation is written to ClickHouse.
4. New facts ("benches 185, goal 225, trains Mon/Wed/Fri") appear in the vault.

**Flow B — Real action via Composio**
1. User: "Draft an email canceling my gym membership" (nice callback to Flow A).
2. Agent uses Composio Gmail tool, drafts/sends.
3. Lint agent later notices: user canceled gym → flags "gym schedule" memory as stale → proposes removal. **This is the demo money shot.**

**Flow C — Auto-lint loop (the headline)**
1. Background worker (Render cron, every 2 min during demo) queries ClickHouse for recent activity.
2. Runs lint rules + an LLM pass over affected vault files.
3. Writes findings to a `lint_findings` table and proposes vault diffs.
4. UI shows the lint feed updating live; user clicks accept; vault file visibly changes.

### Demo script (3 minutes)
1. (20s) Premise: "Agents rot. Ours gardens itself."
2. (60s) Flow A live — intent → generated UI → memory appears.
3. (45s) Flow B live — cancel gym via Composio.
4. (45s) **The reveal:** lint feed fires — "Contradiction detected: gym schedule vs. canceled membership" — show diff, accept, vault updates. Show the ClickHouse query powering it on screen.
5. (10s) Vision slide: "Today on Render; the trajectory is this loop running locally on your phone."

### ClickHouse schema (minimum)

```sql
CREATE TABLE events (
  ts DateTime64(3),
  session_id String,
  user_id String,
  kind Enum('user_msg','agent_msg','tool_call','tool_result','ui_gen','memory_write'),
  tool String DEFAULT '',
  payload String,           -- JSON
  success UInt8 DEFAULT 1,
  latency_ms UInt32 DEFAULT 0
) ENGINE = MergeTree ORDER BY (user_id, ts);

CREATE TABLE memory_entries (
  id String, user_id String, file String, key String,
  value String, source_event String, updated DateTime64(3),
  status Enum('active','stale','retired') DEFAULT 'active'
) ENGINE = ReplacingMergeTree(updated) ORDER BY (user_id, id);

CREATE TABLE lint_findings (
  ts DateTime64(3), user_id String,
  rule Enum('contradiction','staleness','redundancy','failure_pattern','gap'),
  severity Enum('info','warn','fix'),
  description String, proposed_diff String,
  status Enum('open','accepted','rejected') DEFAULT 'open'
) ENGINE = MergeTree ORDER BY (user_id, ts);
```

### Lint rules v0 (ship these five)
1. **Contradiction** — LLM pass: feed pairs of memory entries touching the same key/topic from the last 24h; ask for conflicts. (Demo: gym schedule vs. cancellation.)
2. **Staleness** — SQL: active entries not referenced in any event for N minutes (set N tiny for demo).
3. **Redundancy** — embedding or LLM near-duplicate detection within a file; propose merge.
4. **Failure pattern** — SQL: tool with ≥3 failures in window → propose instruction patch ("Gmail tool requires X; add to skill").
5. **Gap** — LLM: questions the agent answered with low confidence / "I don't know" → propose asking user.

Each finding = `{rule, severity, description, proposed_diff}`. Diffs are plain unified diffs against vault markdown — easy to render, easy to apply.

### Vault format
Plain markdown files (Obsidian-compatible — say this on stage): `profile.md`, `preferences.md`, `projects/gym.md`, `skills/*.md`. The agent loop injects relevant files into context per request (simple keyword/embedding retrieval; keyword is fine for demo).
